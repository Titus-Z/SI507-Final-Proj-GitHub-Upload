# 06. 交互层为什么分成 CLI 和网页

阅读时间建议：20-30 分钟

## 这份文档的目标

读完这一份，你要能解释：

1. 为什么项目不只保留一种交互方式
2. CLI、查询网页、拖拽图网页分别负责什么
3. 为什么查询网站和图网站要拆开
4. 交互层为什么不应该重写业务逻辑

## 1. 交互层不是项目核心，但很重要

这个项目的核心依然是图。

但是一个期末项目如果只有后台逻辑，没有交互入口，展示会很吃亏。

所以项目最终保留了三种入口：

- CLI
- 查询型网页
- 可拖拽图网页

这不是为了花哨，而是因为三者各自适合不同场景。

## 2. CLI 负责什么

CLI 的入口是：

- `main.py`
- `cli.py`

CLI 的优点是：

- 最容易 debug
- 最稳定
- 最适合先验证分析逻辑

在这个项目里，CLI 主要做：

- 搜股票
- 查行业
- 比较两只股票
- 找路径
- 看最中心节点

CLI 的意义在于：先证明系统逻辑是通的。

## 3. 查询型网页负责什么

网页入口是：

- `query_app.py`
- `streamlit_app.py`

这个页面不是为了看一个大图，而是为了把图上的查询变成更直观的界面。

它适合做：

- 股票信息查询
- 行业信息查询
- 股票比较
- 路径查询
- 中心节点查看
- LLM 新闻影响展示
- 新新闻影响评估

你可以把它理解成：

**图分析能力的可视化前端。**

在当前这份课程纯净版里，这个查询网页会自动只显示真正可用的主线功能。

也就是说：

- 核心查询、LLM 历史新闻影响、以及新新闻影响评估一定会显示
- 那些不属于课程主线、而且当前包里并没有提供实现文件的可选功能，不会硬留一个空标签页

这样做的好处是：

- 老师打开网页时不会被无关入口分散注意力
- 课程版交付更干净
- 你也更容易讲清楚“这个网页到底负责什么”

## 4. 为什么可拖拽图网页要单独拆出来

这一步是个非常正确的设计决定。

拖拽图页面入口是：

- `interactive_graph_app.py`

它和查询型网页不应该混在一起，原因很明确：

### 原因 1：目标不同

查询网页的目标是回答明确问题。

可拖拽图页面的目标是：

- 观察整体结构
- 拖动节点
- 缩放和平移
- 看局部团块和桥梁

### 原因 2：认知负担不同

如果把所有功能都堆在一个页面里，用户会同时面对：

- 表格
- 搜索
- 路径查询
- 中心性
- 大图交互

这会让页面非常乱。

### 原因 3：技术栈侧重点不同

查询网页更像表单和数据表。

拖拽图页面更像图可视化。

它们的实现逻辑虽然共享底层图，但表现层思路不同。

所以拆开是合理的。

## 5. 交互层为什么不能重写分析逻辑

这点很关键。

无论是 CLI 还是网页，它们都不应该自己重新处理数据、重新建图、重新实现分析逻辑。

它们应该只做两件事：

1. 接收用户输入
2. 调用现有分析层，然后展示结果

为什么？

因为如果你把业务逻辑复制到多个入口：

- 会很难维护
- 不同入口结果可能不一致
- debug 会非常痛苦

所以项目里做得比较对的一点是：

交互层复用同一套本地数据读取和 analyzer 构建逻辑。

## 6. 为什么这个项目有多种交互模式仍然不算过度设计

课程要求里常见的一条是“至少四种交互模式”。

这个项目其实已经满足：

- 搜索股票
- 浏览行业
- 比较股票
- 查询路径
- 查看中心节点
- 查看新闻影响
- 输入一条新新闻并评估其影响

而这些模式并不是强行凑出来的，而是直接来自图分析层本身。

所以它们不是表面功能，而是真正的分析入口。

## 7. LLM 影响页为什么放在查询网页里

这也是个很合理的放置方式。

因为新闻影响分析本质上是结构化查询结果，不是大图浏览。

它最适合展示成：

- 行业影响汇总表
- 股票影响汇总表
- 文章影响明细表

所以把它放进查询型网页，而不是拖拽图页面，是对的。

同样的道理也适用于“输入一条新新闻 -> 评估其影响”这个功能。

它更像一个受控表单和结果面板，而不是一个需要节点拖拽的大图界面，所以也应该放在查询型网页里。

## 8. 你应该怎么理解三个入口的关系

最好的理解方式是：

### CLI

最基础、最稳、最便于验证。

### 查询型网页

最适合课堂展示和日常操作。

### 拖拽图网页

最适合展示图结构和视觉效果。

它们不是彼此竞争，而是不同层次的入口。

## 9. 你现在应该能讲出来的话

> 项目有三个交互入口：CLI、查询型网页和可拖拽图网页。CLI 最稳定，适合验证分析逻辑；查询网页负责把图查询变成直观界面；拖拽图网页负责展示整体网络结构。它们共享同一套本地数据读取和图分析逻辑，而不是各自重写业务代码，因此既满足多交互模式要求，也保持了架构清晰。

这段话能讲顺，说明你已经理解交互层为什么这样设计。
