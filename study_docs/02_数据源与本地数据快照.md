# 02. 数据源与本地数据快照

阅读时间建议：20-30 分钟

## 这份文档的目标

读完这一份，你应该知道：

1. 项目到底依赖哪些数据源
2. 为什么最后运行时主要读取本地数据
3. `project/data/raw/` 里的文件分别是什么
4. 为什么“数据获取”和“数据分析”被拆开

## 1. 项目用了哪些数据源

这个项目有三个主要数据来源。

### 股票价格

价格数据主要来自 `yfinance`。

你可以把它理解成：

- 用来获取历史日线价格
- 主要服务于股票之间的相关性计算

项目里真正关心的不是每根 K 线的细节，而是：

- 一批股票的价格时间序列
- 由此导出的收益率
- 进一步算出来的相关性

### 行业信息

行业信息来自公司元数据。

项目的目标不是做非常精细的 fundamental analysis，而是需要一个稳定的行业标签，用来：

- 建 `stock -> sector` 边
- 按行业组织股票
- 后续做行业级分析

### 新闻数据

新闻原始数据来自 Alpha Vantage 的 `NEWS_SENTIMENT`。

它的价值不在于“新闻很多”，而在于它给出了结构化字段：

- article 信息
- topics
- ticker_sentiment
- overall sentiment

这使得它天然适合被转成图数据。

## 2. 为什么最后运行时主要读取本地数据

这是项目里一个非常重要的设计判断。

运行分析时，项目默认**不直接打 API**，而是从本地读取缓存好的数据。

原因很现实：

### 原因 1：稳定

如果每次运行都去请求 API，你会遇到：

- 网络问题
- 限流
- 数据变动
- 成本波动

这会让分析体验非常不稳定。

### 原因 2：可重复

老师、助教、你自己三天后再跑一次，最好能看到基本一致的结构。

如果你每次都重新请求最新新闻，图会一直变化，项目就很难验证。

### 原因 3：更容易 debug

出了问题时，你可以先检查本地文件，再检查处理逻辑，而不是一上来怀疑网络或 API。

所以项目最终采用的是：

```text
先下载数据 -> 保存本地 -> 分析时只读本地
```

## 3. `project/data/raw/` 里有什么

这次整理出来的提交包里，已经放了一份本地数据快照。

主要目录是：

- `data/raw/prices/`
- `data/raw/sectors/`
- `data/raw/news/`
- `data/raw/llm_enriched/`

当前这份课程快照大致包含：

- `50` 个价格 CSV
- `50` 个行业 JSON
- `50` 个单 ticker 新闻 JSON
- `41,913` 条合并去重后的新闻文章

### `prices/`

这里放的是每只股票一个 CSV。

例如：

- `AAPL_prices.csv`
- `MSFT_prices.csv`
- `NVDA_prices.csv`

每个文件里大致有这些列：

- `date`
- `open`
- `high`
- `low`
- `close`
- `adjusted_close`
- `volume`

真正后面会被用到的核心列是：

- `date`
- `adjusted_close`

因为计算收益率最关键的是复权收盘价。

### `sectors/`

这里放的是每只股票一个 JSON。

例如：

- `AAPL_sector.json`
- `JPM_sector.json`

主要字段包括：

- `ticker`
- `company_name`
- `sector`
- `industry`

这层数据会直接变成股票节点属性，以及 `stock_sector` 边的来源。

### `news/`

这里至少有一个非常关键的文件：

- `merged_seed_news.json`

它是多个 ticker 新闻合并、去重之后的总新闻库。

后面的新闻处理层主要就从这里读。

这个设计的好处是：

- 不再需要反复请求新闻 API
- 可以用同一份新闻数据做多次实验

### `llm_enriched/`

这个目录里放的是 LLM 离线 enrichment 的结果。

也就是说，新闻先被读取和清洗，再由 OpenAI API 进行结构化影响提取，最后输出成 CSV。

这部分是项目的增强层，不是最底层数据源。

## 4. 项目里谁负责读这些本地文件

正式运行时，最重要的本地读取文件是：

- `local_data_store.py`

它的职责非常清晰：

- 读价格表
- 读行业表
- 读新闻 JSON
- 读 LLM enrichment CSV

你可以把它理解成正式版分析入口的数据门面。

早期的数据获取则在：

- `data_loader.py`
- `seed_data.py`

但最终分析阶段，主要依赖的是本地读取，而不是在线获取。

## 5. 为什么“数据获取”和“数据分析”必须拆开

这个判断非常值得你在答辩时讲出来。

因为这说明你没有把所有事情塞进一个脚本里，而是有明确分层：

### 数据获取层

负责：

- 从外部拉数据
- 保存最新数据
- 写备份

### 数据分析层

负责：

- 读本地数据
- 处理新闻
- 建图
- 做分析
- 做交互展示

这两层一旦拆开，项目的可维护性和可解释性会明显提升。

## 6. 这份数据快照对你意味着什么

你现在已经不用再纠结：

- 数据能不能取到
- API 会不会限流
- 运行时会不会卡在下载

因为提交包里已经带了一份可运行数据快照。

这意味着你之后的学习重点应该从“如何拿数据”转向“这些数据最后被怎么用”。

## 7. 你现在应该形成的认识

如果你真的理解了这份文档，你应该能讲清楚下面这段话：

> 这个项目的数据来源包括股票价格、行业信息和新闻数据，但分析时默认不直接打 API，而是读取本地缓存快照。这样做是为了稳定、可重复和便于调试。价格文件主要用于收益率和相关性计算，行业文件用于构建股票到行业的关系，合并后的新闻文件用于生成主题和股票之间的连接，LLM enrichment 文件则进一步提供新闻对行业和股票影响的结构化结果。

这段话是后面所有代码理解的前提。
