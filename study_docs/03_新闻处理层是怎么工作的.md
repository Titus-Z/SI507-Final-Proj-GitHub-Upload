# 03. 新闻处理层是怎么工作的

阅读时间建议：20-30 分钟

## 这份文档的目标

读完这一份，你应该明白：

1. 原始新闻 JSON 为什么不能直接建图
2. `NewsProcessor` 具体做了什么
3. 为什么会拆成三张中间表
4. `topic_stock` 为什么是后面建图的关键输入

## 1. 原始新闻 JSON 的问题

原始新闻 JSON 看起来信息很多，但对建图来说并不直接可用。

原因有三个。

### 原因 1：结构嵌套

一篇新闻里通常既有：

- 基本文章字段
- `topics`
- `ticker_sentiment`

而且 `topics` 和 `ticker_sentiment` 都是列表。

这意味着一篇文章不是一行平铺数据，而是一个嵌套对象。

### 原因 2：一篇新闻会关联多个 ticker 和多个 topic

如果你直接拿原始 JSON 去建图，会遇到两个问题：

- 很难统计某个 topic 和某只股票到底共同出现了多少次
- 很难计算聚合后的情绪和相关度

### 原因 3：图构建层需要的是关系表，不是原始对象

图最需要的是这种形式的数据：

- 这个股票和这个 topic 之间有多强的关系
- 这个文章和这个 ticker 之间是什么联系

这些都需要先整理成表。

## 2. `NewsProcessor` 的职责

在正式版项目里，负责做这件事的文件是：

- `news_processor.py`

它的核心职责不是“分析新闻内容”，而是：

**把原始新闻 JSON 整理成后面建图和分析可以直接使用的表格。**

这是一个典型的中间层。

它夹在：

- 本地原始新闻文件
- 图构建逻辑

之间。

## 3. 三张核心输出表

这个处理层最重要的设计就是三张表。

### 第一张：`articles`

这张表是文章级别的。

每一行代表一篇新闻。

常见字段包括：

- `article_id`
- `title`
- `url`
- `published_at`
- `source`
- `summary`
- `overall_sentiment_score`
- `overall_sentiment_label`

这张表的作用是：

- 保留新闻本体
- 给后面做文章展示、抽查、LLM enrichment 提供基础信息

### 第二张：`article_tickers`

这张表是文章和 ticker 的关系表。

每一行代表：

- 某一篇文章
- 提到了某一只股票
- 并且有该 ticker 对应的相关度与情绪

核心字段包括：

- `article_id`
- `ticker`
- `relevance_score`
- `ticker_sentiment_score`
- `ticker_sentiment_label`

这张表的作用是：

- 明确文章和股票之间的连接
- 后续筛选某组股票相关新闻时非常有用

### 第三张：`topic_stock`

这张表是最重要的一张。

它不是原始表，而是聚合表。

每一行代表：

- 某个 `ticker`
- 某个 `topic`
- 它们之间聚合之后的关系强度

常见字段包括：

- `ticker`
- `topic`
- `article_count`
- `avg_topic_relevance`
- `avg_ticker_relevance`
- `avg_ticker_sentiment`
- `avg_overall_sentiment`

这张表直接决定了后面 `stock_topic` 边怎么建。

## 4. 为什么一定要有 `article_id`

如果你读代码时看到 `article_id`，不要把它当成随便加的辅助列。

它非常关键。

因为一旦新闻被拆成多张表，你必须要有一个稳定主键把它们重新连起来。

否则：

- 文章表和 ticker 表没法对齐
- LLM enrichment 结果没法回连到原文章
- 网站里文章级影响明细也没法正确展示

所以 `article_id` 是整个新闻层的主键。

## 5. `topic_stock` 是怎么来的

这一步是很多人第一次看时最容易糊涂的地方。

逻辑其实很朴素。

一篇新闻可能有：

- 多个 topic
- 多个 ticker

所以你可以把它看成在生成很多个 `(ticker, topic)` 组合。

然后把这些组合按：

- ticker
- topic

进行分组聚合。

于是就会得到：

- 这个 topic 和这只股票一起出现了多少次
- 平均相关度是多少
- 平均情绪是多少

这正是建 `stock_topic` 边所需要的数据。

## 6. 为什么第一版主图不直接用 article 节点

这是一个你在答辩时很值得讲的设计取舍。

因为文章节点当然可以建，但第一版不适合。

原因是：

- 文章太多
- 图会非常碎
- 路径会很乱
- 展示会很难看

而 topic 节点比 article 节点更适合作为第一版主新闻层，因为它更稳定，也更好解释。

所以这个项目第一版采用的是：

```text
article -> 处理中间层 -> topic_stock -> topic 节点
```

而不是：

```text
article 直接进主图
```

## 7. 新闻处理层在整个项目里的位置

现在你应该能把它放进全流程里了：

```text
本地新闻 JSON
-> NewsProcessor
-> articles / article_tickers / topic_stock
-> NetworkBuilder
```

如果没有这层，图构建会变得非常混乱。

## 8. 你读代码时应该重点看什么

如果你准备去读 `news_processor.py`，建议你重点盯住这几个问题：

1. 文章字段是怎么被提取出来的？
2. `ticker_sentiment` 是怎么展开成关系表的？
3. `topics` 是怎么进入聚合逻辑的？
4. `topic_stock` 的每个聚合字段各代表什么？

只要这四个问题你搞清楚，新闻处理层就算吃透了。

## 9. 你现在应该能讲出来的话

> 新闻原始 JSON 不能直接用来建图，因为它是嵌套结构，而且一篇文章会同时关联多个 ticker 和多个 topic。所以项目先通过 `NewsProcessor` 把新闻整理成三张表：文章表、文章-股票关系表，以及最重要的 `topic_stock` 聚合表。后面图里的 `stock_topic` 边就是由 `topic_stock` 生成的，因此新闻处理层本质上是在把原始新闻转化成图可以直接消费的关系数据。

这句话讲顺了，后面图构建层你就容易理解了。
