# 08. 测试、验证与可信度

阅读时间建议：20-30 分钟

## 这份文档的目标

读完这一份，你要理解：

1. 为什么这个项目必须有测试
2. 项目里哪些层被测试了
3. “测试通过”到底说明了什么
4. 你在展示时应该怎么谈可信度

## 1. 为什么测试在这个项目里很重要

这个项目不是一个单纯的网页，也不是一个单纯的数据脚本。

它是一个多层系统：

- 数据获取
- 本地读取
- 新闻处理
- 图构建
- 图分析
- 交互层
- LLM enrichment

只要其中任意一层结构不稳定，后面的结果都会跟着变形。

所以测试的价值不是“为了看起来专业”，而是为了保证：

- 输入输出格式没坏
- 关键逻辑没有回归
- 图结构仍然符合预期

## 2. 测试覆盖了哪些部分

项目里已经有一组相对完整的测试。

你可以在：

- `project/tests/`

里看到它们。

大致包括：

### 数据层测试

- `test_data_loader.py`
- `test_local_data_store.py`
- `test_local_data_store_llm.py`

这些测试验证的是：

- 文件能不能读到

### 新闻处理层测试

- `test_news_processor.py`

它主要验证：

- 原始 JSON 能不能变成三张表
- 核心列是否存在
- 聚合逻辑是否稳定

### 图构建层测试

- `test_network_builder.py`

它验证的是：

- 节点有没有正确建立
- 边有没有按预期建立
- 图构建逻辑是否满足基本约束

### 图分析层测试

- `test_network_analyzer.py`

它验证：

- 股票查询
- 行业查询
- 股票比较
- 路径查找
- 中心性计算

### CLI / 网页辅助逻辑测试

- `test_cli.py`
- `test_streamlit_app.py`
- `test_interactive_graph_app.py`

这些测试的重点不是“页面长什么样”，而是页面底层辅助逻辑对不对。

### LLM enrichment 测试

- `test_llm_enricher.py`
- `test_llm_news_impact_analyzer.py`

它验证的是：

- 结构化结果能否转换成表
- 输出文件是否写到正确位置
- 核心字段是否存在
- 新输入的一条新闻在评估后是否会被限制到当前图允许的股票、行业、主题范围内

## 3. 测试通过到底说明了什么

你必须对这件事保持清醒。

测试通过不代表：

- 模型一定正确
- 图一定最优
- 新闻影响判断一定准确

测试通过只代表：

**系统在当前设定下，关键逻辑是自洽的，输入输出接口是稳定的。**

这是“工程正确性”，不是“结论绝对正确性”。

这两者必须区分开。

## 4. 项目里哪些东西是“可验证”的

这个项目比较好的地方是，它不是全靠主观解释。

很多部分都是可验证的。

### 价格与行业数据

可以直接打开 CSV / JSON 检查。

### 新闻处理

可以把某篇文章和处理后表格进行对照。

### 图构建

可以检查边是否符合规则来源。

### 图分析

可以用小图手工验证路径和邻居结果。

### LLM enrichment

可以人工抽查文章，验证 sector / ticker / direction 是否大致合理。

## 5. 为什么 LLM 层尤其需要人工抽查

因为 LLM 输出虽然是结构化的，但本质上仍然是模型判断。

所以它比价格相关性更需要抽查。

你在展示时应该这样表达：

- 我们对 LLM 输出做结构化约束
- 我们把结果离线保存，方便抽查
- 我们在展示层只显示本地已生成结果

这会比“模型说了算”要严谨得多。

## 7. 展示时怎么谈“可信度”

你不应该说：

“这个结果一定对。”

你应该说：

“这个系统的关键数据流和逻辑都经过了测试；其中规则型部分具有清晰来源，LLM 部分则通过结构化输出和离线保存来保证可检查性。”

这种说法更专业，也更稳。

## 8. 你现在应该能讲出来的话

> 这个项目的可信度来自两个层面。第一，工程层面上，数据读取、新闻处理、图构建、图分析和 LLM enrichment 都有对应测试，保证输入输出接口和核心逻辑稳定。第二，分析层面上，价格相关性、行业归属和新闻聚合都具有明确来源，而 LLM 输出则采用结构化 schema 并离线落盘，便于人工抽查。因此项目不是纯演示，而是一个可验证的系统。

如果这段话你能讲清楚，测试这一层就不是“附属品”，而是项目质量的一部分。
